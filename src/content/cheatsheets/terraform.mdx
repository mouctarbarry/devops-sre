## Workflow de base

```
terraform init → terraform plan → terraform apply → terraform destroy
```

<TerminalBlock command="terraform init" output="Initialise le backend, télécharge les providers et modules" />

<TerminalBlock command="terraform plan" output="Affiche les changements prévus sans les appliquer" />

<TerminalBlock command="terraform apply" output="Applique les changements (demande confirmation)" />

<TerminalBlock command="terraform apply -auto-approve" output="Applique sans demander confirmation (CI/CD)" />

<TerminalBlock command="terraform destroy" output="Supprime toutes les ressources gérées" />

## Structure de projet

```
infra/
├── main.tf           # Ressources principales
├── variables.tf      # Déclaration des variables
├── outputs.tf        # Valeurs de sortie
├── providers.tf      # Configuration des providers
├── terraform.tfvars  # Valeurs des variables (non commit)
├── backend.tf        # Configuration du state backend
└── modules/
    ├── vpc/
    ├── eks/
    └── rds/
```

  cidr_block        = cidrsubnet(aws_vpc.main.cidr_block, 8, index(["a", "b", "c"], each.key))

## Bonnes pratiques

- **Remote state** : S3 + DynamoDB pour le locking
- **Modules** : réutiliser et versionner l'infrastructure
- **Workspaces** : séparer dev/staging/prod
- **Plan avant apply** : toujours reviewer le plan
- **Versionner les providers** : pinning avec `~>` ou `=`
- **tfvars secrets** : ne jamais commiter, utiliser des variables d'environnement ou un vault
